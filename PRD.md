# PRD: PRD Task Breaker

## 1. 제품 개요

개발자가 PRD를 입력하면 즉시 개발 착수 가능한 구조화된 산출물을 생성하는 AI 도구.

## 2. 목표

- PRD → 구조화된 개발 태스크 자동 변환
- 기획-개발 간 커뮤니케이션 비용 절감
- MVP 품질의 API/DB 설계 초안 자동 생성

## 3. 사용자

- 주요: 백엔드/풀스택 개발자
- 부수: 기획자, 스타트업 창업자

## 3-1. 사용자 페르소나

### 페르소나 A — 백엔드/풀스택 개발자 (주요)
- **상황**: 스타트업 또는 소규모 팀, 기획서를 받으면 즉시 개발을 시작해야 하는 압박
- **문제**: PRD를 보고 API/DB 설계·태스크 분해를 직접 해야 해서 수 시간 소요; 기획 의도를 놓쳐 재작업 빈번
- **목표**: PRD 입력 후 30초 내에 개발 착수 가능한 산출물(API 초안, DB 스키마, TODO) 확보
- **행동 패턴**: PRD 붙여넣기 → 9개 섹션 결과 검토 → 필요 섹션만 PDF로 저장 후 팀에 공유

### 페르소나 B — 스타트업 기획자 / 창업자 (부수)
- **상황**: 개발 지식이 제한적이나 개발팀과 구체적인 범위 협의가 필요
- **문제**: 기획서를 넘겨도 개발팀이 구체화하는 과정에서 병목이 생기고 일정이 밀림
- **목표**: 기획서를 입력하면 개발 범위·우선순위·불확실 항목이 즉시 구조화되어 소통 비용 감소
- **행동 패턴**: 기획서 초안 작성 → 분석 실행 → 불확실 항목 섹션으로 개발팀과 논의 포인트 정리

## 3-2. 시장 및 경쟁 환경

### 타겟 시장
- **1차 타겟**: 소규모 팀·스타트업의 백엔드/풀스택 개발자 (5인 이하 팀)
- **2차 타겟**: 개발팀과 직접 소통해야 하는 비기술 기획자·창업자
- **시장 맥락**: AI 코딩/개발 생산성 도구 시장은 2024~2025년 급성장 중. GitHub Copilot·Cursor 등이 코드 작성 단계를 커버하는 반면, PRD → 태스크 분해라는 **계획 단계** 자동화 도구는 아직 공백이 큼.

### 경쟁 솔루션 분석

| 솔루션 | 강점 | 한계 | PRD Task Breaker 대비 |
|--------|------|------|----------------------|
| **Jira** | 강력한 프로젝트 관리, 팀 협업 | PRD를 직접 분석하지 않음; 태스크는 사람이 수동 입력 | AI 자동 분해 → 입력 즉시 산출물 |
| **Linear** | 빠른 이슈 트래킹, 개발자 친화적 | PRD → 이슈 자동 변환 기능 없음 | 30초 내 API/DB 초안까지 동시 생성 |
| **Notion AI** | 유연한 문서 편집 + AI 요약 | 태스크 구조화·API 설계는 직접 프롬프트 필요; 결과 일관성 낮음 | 9개 섹션 고정 스키마로 일관된 출력 |
| **ChatGPT/Claude 직접 사용** | 높은 자유도 | 프롬프트 작성 부담; 결과 포맷 불일치; 이력 관리 없음 | 전용 프롬프트 + 구조화 스키마 + 히스토리 저장 |

### 차별화 포인트 요약
1. **PRD 특화 스키마**: 범용 AI 도구와 달리 9개 섹션(API/DB/테스트/릴리즈 체크리스트 포함) 고정 출력
2. **즉시 사용 가능한 산출물**: 분석 결과를 팀 공유용 PDF로 바로 내보내기
3. **이력 관리**: 과거 분석 결과 저장 및 재조회
4. **낮은 진입 장벽**: 별도 계정·프로젝트 설정 없이 URL 접속 후 즉시 사용

## 4. 핵심 기능

### F-01: PRD 입력

- 텍스트 영역에 PRD 직접 입력 (50~50,000자)
- 샘플 PRD 불러오기 버튼

### F-02: AI 분석 실행

- 분석 버튼 클릭 → POST /api/v1/analysis
- 로딩 상태 표시

### F-03: 결과 표시 (9개 섹션)

1. **기능 목록** (Feature List): 이름, 설명, 우선순위
2. **유저 스토리** (User Stories): 역할, 행동, 이점, 인수 기준
3. **TODO 항목** (TODO Breakdown): 태스크, 카테고리, 우선순위, 예상 공수
4. **API 초안** (API Draft): Method, Path, 요청/응답 바디
5. **DB 초안** (DB Draft): 테이블, 컬럼, 타입
6. **테스트 체크리스트** (Test Checklist): 항목, 카테고리
7. **릴리즈 체크리스트** (Release Checklist): 항목, 카테고리
8. **불확실 항목** (Uncertain Items): 추가 논의 필요 사항
9. **README 초안** (README Draft): 프로젝트 README 마크다운

### F-04: 에러/빈 상태 처리

- 입력 없음, AI 오류, 서버 오류 각각 안내 메시지

### F-05: PDF 내보내기

- 분석 결과를 PDF로 저장 (섹션별 선택 가능)
- 한글 폰트 지원 (Noto Sans KR TTF)

## 5. 성공 지표 (KPI)

| 지표 | 목표값 | 측정 방법 |
|------|--------|-----------|
| 분석 완료율 | 입력 대비 90%+ | 서버 응답 코드 200 비율 |
| 응답 시간 P95 | 30초 이내 | 서버 로그 처리 시간 |
| PDF 내보내기 사용률 | 분석 완료 건의 30%+ | PDF 다운로드 버튼 클릭 수 |
| 히스토리 재조회율 | 전체 세션의 40%+ | `/api/v1/analysis/{id}` 호출 수 |
| 오류율 | 전체 요청의 5% 미만 | 서버 4xx/5xx 응답 비율 |

## 6. 비기능 요구사항

- 응답 시간: 30초 이내 (AI 호출 포함)
- Docker Compose 단일 명령 실행
- 테스트 커버리지: 주요 서비스 로직 80%+

## 7. 기술 스택

- Frontend: Vue 3, TypeScript, Vite
- Backend: Spring Boot 3.2, Java 17, Gradle
- DB: H2 (dev), PostgreSQL (prod)
- AI: Anthropic Claude API
- CI: GitHub Actions
